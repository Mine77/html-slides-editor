HTML Should Be the Source Format for Digital Content

235|

236| Slides, documents, videos, interactive explainers — the next generation of digital content 237| should be authored in HTML. Not because it's trendy, but because it's the only format that 238| is simultaneously human-readable, machine-writable, and universally renderable. 239|

240| 241|
242| 243| 244|

1. Better Expressiveness

245| 246|

For humans: beyond flat text

247|

248| PowerPoint gave us slides. Word gave us documents. Each tool created its own island format, 249| and each format carries the limitations of the era it was born in. HTML, by contrast, is a 250| living standard — it has evolved over 30 years to support everything from 251| simple text to embedded video, interactive charts, animated transitions, and responsive 252| layouts that adapt to any screen. 253|

254|

255| When your source file is HTML, a slide deck can contain an embedded data visualization that 256| updates in real time. A course module can include an interactive code playground. A research 257| paper can embed a runnable experiment. No export step, no conversion loss, no "this feature 258| is not supported in this format." 259|

260| 261|

For agents: semantic, structured, and programmable

262|

263| LLMs don't see pixels — they see text. HTML is the richest text-based format that still 264| carries explicit structure: headings, lists, tables, sections, and semantic 265| tags like <article>, <figure>, and 266| <nav>. An agent can read an HTML document and understand not just 267| what the content says, but how it is organized. 268|

269|

270| Compare this to a .pptx file: to an agent, it's a zip archive of XML blobs 271| with coordinate-based positioning. To modify a single bullet point, the agent must parse 272| OPC packaging, navigate slide1.xml, locate the right 273| <a:p> element among dozens, and reconstruct the XML without breaking 274| the slide layout. With HTML, the same change is a one-line text edit. 275|

276| 277| 278|

2. No Vendor Lock-in

279| 280|

281| The dominant content formats today — .pptx, .docx, 282| .key, .aep — are controlled by single vendors. They can change 283| the spec, discontinue the tool, or raise the price, and your content goes along for the 284| ride. 285|

286|

287| HTML is an open standard maintained by the W3C. Every browser renders it. Every operating 288| system supports it. No single company can deprecate it. Your content, authored in HTML today, 289| will be readable in 2040 without asking anyone's permission. 290|

291| 292|

For the agent-human collaboration era, this matters even more:

293|
    294|
  • 295| Agents can generate HTML natively. No need for specialized libraries or 296| reverse-engineered file format specs. The agent writes text, and the text is 297| the content. 298|
    • 299|
  • 300| Humans can edit HTML in any tool. A browser, a text editor, an IDE, 301| a visual editor — all of them work. There is no "you need this specific app to open 302| this file" problem. 303|
    • 304|
  • 305| Version control is natural. HTML is plain text. git diff 306| shows you exactly what changed, line by line. Try diffing two .pptx files. 307|
    • 308|
309| 310| 311|

3. Composable with the Web Platform

312|

313| HTML doesn't exist in isolation. It sits on top of CSS and JavaScript — the most widely 314| deployed programming platform in history. This means an HTML source file can: 315|

316|
    317|
  • Use CSS for visual design without embedding styles in the content structure.
    • 318|
  • Use JavaScript for interactivity — animations, data binding, user input.
    • 319|
  • Fetch live data from APIs and render it inline.
    • 320|
  • Be embedded in other HTML pages, shared via URL, or printed to PDF.
    • 321|
322|

323| No other content format offers this level of composability. A PPTX file is a dead artifact. 324| An HTML file is a living application. 325|

326| 327| 328|

4. Accessibility by Default

329|

330| HTML has built-in accessibility semantics. Screen readers understand 331| <h1> through <h6>, <nav>, 332| <table>, alt attributes on images, and ARIA roles. 333| When you author content in HTML, accessibility is a natural byproduct of good structure. 334|

335|

336| Proprietary formats require separate accessibility passes — and most authors skip them. 337| The result: millions of PowerPoint presentations that are invisible to screen readers. 338| HTML makes the accessible path the easy path. 339|

344| We are entering an era where AI agents will author, edit, and remix content alongside 345| humans. This collaboration requires a format that both parties can read and write 346| fluently. HTML is the only candidate: 347|

The Missing Piece: Tooling

460| 461|

462| If HTML is such a great source format, why isn't everyone using it already? Because the 463| tooling gap is real. 464|

465|

466| Writing raw HTML by hand is tedious. Existing visual editors (Figma, Canva, Google Slides) 467| export to HTML but don't use it as their source format. Web IDEs understand HTML but aren't 468| designed for content creation workflows — they think in terms of components and builds, not 469| slides and chapters. 470|

471|

472| What we need is a new class of tools that: 473|

474|
    475|
  • Use HTML as the native source format — not an export target.
    • 476|
  • Provide visual editing for humans — drag, resize, style — without hiding the HTML.
    • 477|
  • Expose a structured contract for agents — so AI can generate, verify, and edit content programmatically.
    • 478|
  • Treat the source file as the single source of truth — no sidecar metadata, no hidden state.
    • 479|
480| 481| 482|
483|

484| This is what we're building at StarryKit. 485|

486|

487| Our first product, Starry Slides, uses a single HTML file as the source of truth for an 488| entire slide deck. An agent can generate a deck from a contract, verify it against a 489| schema, and open it in a visual editor — all without converting between formats. The 490| human edits visually; the agent reads and writes the same HTML. No lock-in, no conversion 491| loss, no gap between what the human sees and what the agent understands. 492|

493|

494| We believe this pattern — HTML as source, contract as schema, editor as interface — 495| will extend beyond slides to documents, storyboards, dashboards, and formats we haven't 496| imagined yet. 497|
