# Feedback Backend Design

## Endpoint

`POST /feedback`

Accepts a JSON `FeedbackPayload` from the CLI, editor, or skill feedback flows.

Required fields:

- `surface`: `cli`, `editor`, or `skill`
- `category`: `bug`, `ux`, `feature`, or `docs`
- `summary`: short user-facing title
- `description`: detailed report
- `timestamp`: ISO timestamp generated by the client

Optional fields include `expected`, `reproSteps`, `command`, `deckContext`, `verifySummary`, version fields, `nodeVersion`, `os`, and opt-in `attachments`.

Successful response:

```json
{
  "ok": true,
  "id": "github-issue-number-or-backend-id",
  "url": "https://github.com/StarryKit/starry-slides/issues/123"
}
```

Failure response:

```json
{
  "ok": false,
  "error": "Validation failed"
}
```

## Validation

The backend should reject requests that do not match the shared schema. Validate enum fields exactly, require non-empty `summary` and `description`, require an ISO-like `timestamp`, and limit string sizes before any downstream processing.

Recommended limits:

- `summary`: 140 characters
- `description`: 8,000 characters
- optional text fields: 4,000 characters each
- `attachments`: disabled by default or capped by count and byte size when enabled

Unknown fields should be ignored or stripped so clients cannot inject issue metadata or backend options.

## Rate Limiting

Rate limit by IP address and by a coarse anonymous fingerprint when available. V1 can start with a conservative token bucket such as 5 requests per 10 minutes per IP, with a higher internal allowlist for development and support workflows.

The endpoint should return HTTP `429` with a short JSON error when limited. Do not create GitHub issues for limited requests.

## Sanitization

Normalize all text fields as plain text. Strip control characters except standard newlines and tabs, trim leading and trailing whitespace, and escape content when rendering issue markdown.

Attachments must be opt-in. Only persist or forward an attachment when `consented` is `true`, the `type` is allowed, and the content passes size and format checks.

## GitHub Issue Creation

The backend owns GitHub credentials. Clients never receive or manage GitHub tokens.

On valid requests, the backend creates an issue in `StarryKit/starry-slides` using a GitHub App installation token or a fine-scoped service token stored in backend secrets.

Suggested issue title:

```text
[Feedback][<category>][<surface>] <summary>
```

Suggested labels:

- `feedback`
- `feedback:<category>`
- `surface:<surface>`

The issue body should include the description, expected behavior, repro steps, command, deck context, verification summary, app and CLI versions, Node version, OS, timestamp, and attachment summaries. Keep safe metadata in a structured section so maintainers can scan reports consistently.

If GitHub creation fails after validation, return HTTP `502` with `{ "ok": false, "error": "Issue creation failed" }` and log the backend error with request correlation data.

## Privacy Model

Default collection is safe metadata only: category, summary, description, surface, version information, Node version, OS summary, timestamp, and optional deck context counts or file names supplied by the client.

Do not collect deck contents, screenshots, exported files, source HTML, environment variables, path-expanded home directories, or logs by default.

Attachments are opt-in and must carry explicit consent through `attachments[].consented === true`. The UI and skill flows should explain what will be sent before including any attachment. The backend should drop non-consented attachments even if a client sends them.
